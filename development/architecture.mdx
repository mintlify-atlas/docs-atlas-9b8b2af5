ALESPI Solutions Web uses a shell-and-partials architecture: a single `index.html` acts as the application shell, while each section of the page lives in its own HTML file under `src/`. JavaScript fetches and injects those partials at runtime, then binds all interactive behavior once the DOM is fully populated.

## Shell and partials pattern

`index.html` contains no visible content of its own. It defines the page skeleton — `<head>` metadata, stylesheet links, placeholder `<div>` elements, and the script tag — but every piece of rendered content is loaded from a separate file in the `src/` directory.

Placeholder elements use a `data-include` attribute whose value is the relative path to the partial file:

```html
<div data-include="src/navbar.html"></div>
<div data-include="src/hero.html"></div>
<div data-include="src/footer.html"></div>
```

When `loadPartials()` runs, it queries all `[data-include]` elements, fetches each file in parallel, and replaces the placeholder's `innerHTML` with the response text.

## Tab navigation pattern

The site uses a custom tab system built on three HTML conventions:

- **`.tab-btn[data-tab]`** — navigation buttons in the navbar. The `data-tab` value identifies which content panel to show.
- **`[data-tab-content]`** — content wrapper `<div>` elements in `index.html`. Each tab panel has a matching `data-tab-content` value and starts with the class `hidden`, except `inicio` which is `active`.
- **`[data-open-tab]`** — anchor links inside partials that trigger a tab switch programmatically (for example, a "contact us" link inside a service description).

```html
<!-- In navbar.html (after partial injection) -->
<button class="tab-btn" data-tab="servicios">Servicios</button>

<!-- In index.html -->
<div data-tab-content="servicios" class="tab-content hidden">
  <div data-include="src/servicios.html"></div>
</div>

<!-- In any partial — opens the contact tab when clicked -->
<a href="#" data-open-tab="contacto">Contáctanos</a>
```

## Initialization sequence

The entire setup runs inside a single `DOMContentLoaded` listener. Because partials are fetched asynchronously, `loadPartials()` is `await`ed before any other function runs — the navbar, form, and observer functions all depend on the injected HTML being present in the DOM.

```js
document.addEventListener('DOMContentLoaded', async function() {
    await loadPartials();       // 1. Fetch and inject all src/*.html partials
    initializeTabs();           // 2. Bind tab button and data-open-tab click events
    initializeHamburger();      // 3. Bind hamburger toggle for mobile nav
    initializeContactForm();    // 4. Bind form submit + validation
    initializeAnchorScroll();   // 5. Smooth-scroll for in-page anchor links
    initializeStatsObserver();  // 6. Count-up animation for .stat-number elements
    initializeCardObserver();   // 7. Fade-in animation for .service-card, .valor-card
    initializeNavbarShadow();   // 8. Deepen navbar shadow on scroll
});
```

<Note>
  Steps 2–8 must run after `loadPartials()` resolves. If any `initialize*` function runs before the partials are injected, it will find no matching DOM elements and silently do nothing — or worse, bind events to elements that do not exist yet.
</Note>

## The `loadPartials()` function

`loadPartials()` collects every element with a `data-include` attribute, fires all `fetch()` calls concurrently with `Promise.all`, and writes the response HTML into each placeholder.

```js
async function loadPartials() {
    const partialTargets = document.querySelectorAll('[data-include]');

    await Promise.all(Array.from(partialTargets).map(async (target) => {
        const partialPath = target.getAttribute('data-include');

        try {
            const response = await fetch(partialPath);

            if (!response.ok) {
                throw new Error(`No se pudo cargar ${partialPath}`);
            }

            target.innerHTML = await response.text();
        } catch (error) {
            console.error(error);
            target.innerHTML = `<div class="p-6 text-red-600">Error al cargar ${partialPath}</div>`;
        }
    }));
}
```

If a fetch fails (file not found, server error), the placeholder is replaced with a red error message rather than leaving an empty element.

## The `activateTab()` function

`activateTab()` handles all visual state changes when switching between tabs. It guards against re-activating the currently active tab, strips `tab-active` from all buttons, hides all content panels, then shows the requested one and restarts its CSS animations.

```js
function activateTab(tabName) {
    const tabButtons = document.querySelectorAll('.tab-btn');
    const tabContents = document.querySelectorAll('.tab-content');
    const targetButton = document.querySelector(`.tab-btn[data-tab="${tabName}"]`);
    const activeContent = document.querySelector(`[data-tab-content="${tabName}"]`);

    if (!targetButton || !activeContent || targetButton.classList.contains('tab-active')) {
        return;
    }

    tabButtons.forEach((btn) => btn.classList.remove('tab-active'));
    tabContents.forEach((content) => content.classList.add('hidden'));

    targetButton.classList.add('tab-active');
    activeContent.classList.remove('hidden');
    restartAnimations(activeContent);
    window.scrollTo({ top: 0, behavior: 'smooth' });
}
```

## Data flow: page load to interactive page

The following sequence describes what happens from the moment a browser requests the site to the moment the user can interact with it:

<Steps>
  <Step title="HTML shell is parsed">
    The browser parses `index.html`. All `[data-include]` placeholders and `[data-tab-content]` panels are in the DOM, but the placeholders are empty and all tab panels except `inicio` have class `hidden`.
  </Step>
  <Step title="script.js executes">
    The browser downloads and executes `script.js`. The `DOMContentLoaded` listener is registered and fires immediately because parsing is complete.
  </Step>
  <Step title="Partials fetched in parallel">
    `loadPartials()` queries all `[data-include]` elements and fires one `fetch()` per partial simultaneously. The browser may serve these from cache or the local server.
  </Step>
  <Step title="HTML injected into placeholders">
    As each fetch resolves, its response text is written into the matching placeholder's `innerHTML`. After all promises settle, the full DOM — including the navbar, hero, all sections, and footer — is in place.
  </Step>
  <Step title="Event listeners and observers bound">
    The eight `initialize*()` functions run in sequence. They query the now-populated DOM, attach click listeners, set up `IntersectionObserver` instances, and apply initial inline styles.
  </Step>
  <Step title="Page is interactive">
    The user sees the `inicio` tab content. Clicking any tab button calls `activateTab()`, which swaps visibility and restarts animations.
  </Step>
</Steps>

## Why the site must be served over HTTP

`loadPartials()` uses the browser's `fetch()` API to load the partial files. Browsers enforce the same-origin policy and block `fetch()` requests made from a `file://` URL to local files. Opening `index.html` directly from the filesystem will result in failed fetches and empty sections.

<Warning>
  Always use a local HTTP server when developing. Run `npm run serve` to start `live-server` on port 8080, or use any equivalent tool such as the VS Code Live Server extension or Python's `http.server`.
</Warning>
