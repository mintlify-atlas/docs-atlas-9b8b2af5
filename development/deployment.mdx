ALESPI Solutions Web is a static site with no server-side logic. Deployment consists of two steps: running the Tailwind build to produce `output.css`, then uploading a specific set of files to any HTTP-capable host. This page walks through the build step and deployment options for the most common platforms.

## Files required for deployment

Only a subset of the repository needs to be uploaded. Build tooling, source files, and `node_modules` stay local.

```
ALESPI-Solutions-Web/
├── index.html       ← required
├── output.css       ← required (generated by build step)
├── script.js        ← required
├── images/          ← required (favicon and any image assets)
└── src/             ← required (all HTML partials loaded at runtime)
    ├── navbar.html
    ├── hero.html
    ├── valores.html
    ├── servicios.html
    ├── expertiz.html
    ├── estadisticas.html
    ├── credenciales.html
    ├── contacto.html
    └── footer.html
```

Do not upload `input.css`, `tailwind.config.js`, `postcss.config.js`, `package.json`, or `node_modules/`.

## Build step

Before deploying, compile the Tailwind stylesheet into a minified `output.css`:

```bash
npm install
npm run build
```

`npm run build` runs `tailwindcss -i ./input.css -o ./output.css --minify`. The resulting `output.css` is what `index.html` loads via `<link rel="stylesheet" href="output.css">`.

<Warning>
  Never deploy without running `npm run build` first. If `output.css` is missing or stale, the site will load without styles. Never edit `output.css` directly — always edit `input.css` and rebuild.
</Warning>

## Critical hosting requirement

`script.js` uses the browser `fetch()` API to load HTML partials from the `src/` directory. Browsers block `fetch()` requests made from `file://` URLs due to the same-origin policy.

<Warning>
  You must host the site over HTTP or HTTPS. Opening `index.html` directly in a browser (`file://`) will cause all sections to fail to load. Every platform listed below satisfies this requirement automatically.
</Warning>

## Deployment platforms

<Tabs>
  <Tab title="Netlify">
    Netlify supports both drag-and-drop upload and continuous deployment from a Git repository.

    ### Option A: drag and drop

    <Steps>
      <Step title="Build locally">
        ```bash
        npm install && npm run build
        ```
      </Step>
      <Step title="Upload the deployment folder">
        Go to [app.netlify.com](https://app.netlify.com), open **Sites**, and drag the project root folder (containing `index.html`, `output.css`, `script.js`, `images/`, and `src/`) onto the drop zone.

        Netlify will publish the site immediately and assign a `.netlify.app` subdomain.
      </Step>
    </Steps>

    ### Option B: Git integration

    <Steps>
      <Step title="Commit output.css">
        Commit `output.css` to your repository so Netlify does not need to run a build command.

        Alternatively, set the Netlify **Build command** to `npm run build` and Netlify will build on each deploy.
      </Step>
      <Step title="Connect the repository">
        In Netlify, click **Add new site → Import an existing project** and select your GitHub repository.
      </Step>
      <Step title="Configure build settings">
        | Setting | Value |
        |---|---|
        | Build command | `npm run build` (or leave blank if `output.css` is committed) |
        | Publish directory | `.` (project root) |
      </Step>
      <Step title="Deploy">
        Click **Deploy site**. Netlify builds and publishes. Every push to the configured branch triggers a new deploy automatically.
      </Step>
    </Steps>

    <Tip>
      To accept real contact form submissions, enable **Netlify Forms** by adding `netlify` or `data-netlify="true"` to the `<form>` element in `src/contacto.html`. Netlify will intercept submissions and forward them to your dashboard without any backend code.
    </Tip>
  </Tab>
  <Tab title="Vercel">
    <Steps>
      <Step title="Build locally">
        ```bash
        npm install && npm run build
        ```
      </Step>
      <Step title="Import the repository">
        Go to [vercel.com/new](https://vercel.com/new) and import your GitHub, GitLab, or Bitbucket repository.
      </Step>
      <Step title="Configure project settings">
        On the configuration screen, set:

        | Setting | Value |
        |---|---|
        | Framework preset | **Other** (no framework) |
        | Build command | `npm run build` |
        | Output directory | `.` (project root) |
        | Install command | `npm install` |
      </Step>
      <Step title="Deploy">
        Click **Deploy**. Vercel builds the project and publishes it to a `.vercel.app` subdomain. Subsequent pushes to the production branch deploy automatically.
      </Step>
    </Steps>

    <Note>
      Vercel's edge network serves files over HTTPS by default. The `fetch()` calls in `script.js` will work correctly without any additional configuration.
    </Note>
  </Tab>
  <Tab title="GitHub Pages">
    GitHub Pages serves the contents of a branch or directory directly over HTTPS.

    <Steps>
      <Step title="Build and commit output.css">
        GitHub Pages does not run build commands, so you must commit `output.css`:

        ```bash
        npm install && npm run build
        git add output.css
        git commit -m "build: compile output.css for deployment"
        git push
        ```
      </Step>
      <Step title="Enable GitHub Pages">
        In your repository, go to **Settings → Pages**.

        Under **Source**, choose either:
        - **Deploy from a branch** → select `main` (or your default branch) and the root (`/`) directory.
        - **Deploy from a branch** → select `gh-pages` if you prefer a dedicated deployment branch.
      </Step>
      <Step title="Wait for deployment">
        GitHub Actions will build and publish the Pages site. The URL will be `https://<username>.github.io/<repository>/`.
      </Step>
    </Steps>

    <Warning>
      GitHub Pages serves the repository root. All paths in `index.html` (`src/navbar.html`, `output.css`, `script.js`) are relative, so they resolve correctly. If you ever move `index.html` into a subdirectory, you will need to update those paths.
    </Warning>
  </Tab>
  <Tab title="Any HTTP server">
    Any web server capable of serving static files works. The only requirement is that files are served over HTTP or HTTPS.

    ### Apache

    Copy the deployment files into your Apache `DocumentRoot` (commonly `/var/www/html/`):

    ```bash
    npm run build
    rsync -av --exclude='node_modules' --exclude='input.css' \
      --exclude='tailwind.config.js' --exclude='postcss.config.js' \
      --exclude='package.json' \
      ./ user@yourserver:/var/www/html/alespi/
    ```

    No `.htaccess` configuration is needed — the site uses only static files and relative paths.

    ### Nginx

    Add a server block pointing to the directory containing your files:

    ```nginx
    server {
        listen 80;
        server_name yourdomain.com;
        root /var/www/html/alespi;
        index index.html;

        location / {
            try_files $uri $uri/ =404;
        }
    }
    ```

    ### Python (local testing)

    For quick local verification of a production build:

    ```bash
    npm run build
    python3 -m http.server 8080
    ```

    Open `http://localhost:8080` in your browser.

    ### live-server (development)

    The project includes `live-server` as a devDependency. It reloads the browser automatically when files change:

    ```bash
    npm run serve   # Starts live-server on port 8080
    npm run preview # Builds CSS then starts live-server
    ```
  </Tab>
</Tabs>

## Contact form considerations

The contact form in `src/contacto.html` is entirely frontend. `initializeContactForm()` validates the fields and shows a success notification, but does not transmit any data.

To accept real form submissions you have three options:

<CardGroup cols={3}>
  <Card title="Form service" icon="paper-plane">
    Use **Formspree**, **Getform**, or a similar service. Point the form's `action` attribute to the service endpoint and change the method to `POST`. No backend code required.
  </Card>
  <Card title="Platform forms" icon="server">
    **Netlify Forms** and **Vercel** both offer built-in form handling. Add `data-netlify="true"` (Netlify) or use a Vercel serverless function to process submissions.
  </Card>
  <Card title="Serverless function" icon="bolt">
    Write a serverless function (AWS Lambda, Cloudflare Workers, Vercel Edge Functions) that receives the form data and forwards it via email or stores it in a database.
  </Card>
</CardGroup>

## Environment summary

| Requirement | Reason |
|---|---|
| HTTP or HTTPS hosting | `fetch()` in `script.js` is blocked on `file://` |
| `output.css` present | `index.html` links to this file for all styles |
| `src/` directory present | All section partials are loaded at runtime |
| `images/` directory present | Favicon and any image assets referenced in partials |
